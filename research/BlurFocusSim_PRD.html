§02The Problem

Real-time, physics-accurate, broadcast-quality spatial visualization is currently impossible without dedicated $500K+ hardware racks. Hawkeye renders ball trajectories after the fact. Weta renders Pandora in hours per frame. Game engines render generic worlds with no physics identity. Nobody packages pre-solved physics + live playback + user editing on consumer hardware.

THE GAP

Three industries solve fragments. None solve the whole:

  • VFX (Pixar / Weta) — broadcast quality, but offline-only, hours per frame
  • Game Engines (UE / Unity) — real-time, but no physics-baked identity
  • Sports Broadcast (Hawkeye) — physics-accurate, but read-only and proprietary

No product delivers all three: real-time, physics-accurate, live-editable.

§03The Solution

A hybrid baked-live render pipeline that pre-solves expensive physics on the NPU offline, bakes the result as a 4K AV1 video + curve-keyed DataTable, then plays it back at runtime with hardware video decode (0.3ms/frame) and adds only view-dependent effects on top (Fresnel, specular, Lumen rim — total <3ms). DLSS 4 Multi-Frame Generation multiplies the output 4×. Result: 4K @ 120fps with broadcast-quality physics simulation.

// Sample one row of a curve atlas as a 1D texture
float SampleCurve(Texture2D CurveAtlas, SamplerState Samp, float CurveIndex, float t)
{
    float2 UV = float2(saturate(t), (CurveIndex + 0.5) / NumCurves);
    return CurveAtlas.SampleLevel(Samp, UV, 0).r;
}

§11Baked Look · PostProcess + Reflection + LUT

The Video Bake Pipeline (§10) handles motion. The Curves Architecture (§09) handles parameters. The Baked Look layer handles everything else that defines visual identity: color grading, blur, focus pull, reflection environment, tone mapping, material composition. All baked offline, all costing <0.5ms per frame at runtime.

THE INSIGHT

Real-time path tracers spend 8–15ms per frame computing reflections, GI, and tone curves. But these are fixed for any scene that doesn't change lighting topology. Pre-bake them once into PostProcessVolume settings, sphere/box ReflectionCapture cubemaps, and a 256×16 LUT texture — runtime cost collapses to lookup speed. Same quality, ~30× cheaper.

§13Live Python · MCP Execution Pipeline

The single biggest workflow advantage of this stack is that Python code executes inside the running Unreal Editor in real time via the UnrealClaudeMCP plugin. No restart. No recompile. No save-and-reload. You type Python, it runs in UE's embedded interpreter against the live editor state, and returns stdout/stderr over a length-prefixed JSON TCP protocol.

WHY THIS MATTERS

Traditional UE iteration: edit C++ → recompile (90s) → restart editor (45s) → reproduce state (variable). Total: 2-5 minutes per iteration.

MCP iteration: type Python → execute live (~50ms). Total: instant. Used throughout this entire project — every script in §07 was iterated via this loop.

# Built into ai_command_panel.py — runs in a daemon thread
def _build_watch_thread(self):
    def watch():
        last_mtimes = {}
        while True:
            changed = []
            for root, _, files in os.walk(src_dir):
                for f in files:
                    if not (f.endswith(".cpp") or f.endswith(".h")): continue
                    path = os.path.join(root, f); mtime = os.path.getmtime(path)
                    if last_mtimes.get(path) and last_mtimes[path] != mtime:
                        changed.append(f)
                    last_mtimes[path] = mtime
            if changed:
                self._trigger_build()                       # subprocess Build.bat
                self._send_raw("unreal.SystemLibrary.execute_console_command("
                               "None, 'HotReload BlurFocus')")
            time.sleep(8)
    threading.Thread(target=watch, daemon=True).start()

// Camera Follow Target · The Cinematography Secret

The camera tracking the ball across all three sports uses the same curve-driven follow target pattern. Three curves drive the entire camera director — edit them live in the UE Curve Editor and cinematography changes in real time. 

// BP_BallChaseCam · per-frame follow target with predictive lookahead
void UCameraDirector::Tick(float DeltaTime)
{
    const FVector BallPos  = Ball->GetActorLocation();
    const FVector BallVel  = Ball->GetVelocity();
    const float   Speed    = BallVel.Size() / MaxSpeed; // normalize 0..1

    // Three curves drive the entire cinematography
    const float   LookAhead  = LookAheadCurve->GetFloatValue(Speed);
    const float   FollowDist = FollowOffsetCurve->GetFloatValue(Speed);
    const float   Smooth     = SmoothingCurve->GetFloatValue(DeltaTime);

    // Predict where the ball will be — camera leads the action
    const FVector PredictedPos = BallPos + BallVel * LookAhead;
    const FVector DesiredLoc   = PredictedPos + OrbitDir * FollowDist;

    // Curve-smoothed lerp — no jitter on direction changes
    CurrentCamLoc = FMath::Lerp(CurrentCamLoc, DesiredLoc, Smooth);
    Camera->SetWorldLocation(CurrentCamLoc);
    Camera->SetFieldOfView(FOVCurve->GetFloatValue(Speed));   // FOV pulses with speed

    // Focus pull — depth-of-field tracks the ball
    PostProcess->Settings.DepthOfFieldFocalDistance = (CurrentCamLoc - BallPos).Size();
}

// The Rocket League Signature Trick · Explained

WHY ROCKET LEAGUE FEELS DIFFERENT

In a normal third-person game, the SpringArm attaches to the player and the camera tracks the player. This is why the camera always feels like it's "watching the action" rather than following you specifically. We replicate this for free by attaching BP_BallChaseCam to the SimSphereActor in Rocket mode — same primitive, inverted target.

// The Magnus Effect · Physics on the Projectile

A spinning ball moving through a fluid (air) experiences a lateral force perpendicular to both its velocity and its spin axis. This is what makes a topspin forehand dip violently, a slice serve curve sideways, and a David Beckham free kick bend around a wall. It's the entire reason SPIN feels like real tennis/golf/soccer instead of arcade physics. 

// Added to AProjectile::Tick — the Magnus force calculation
// F_magnus = (1/2) × ρ × C_L × A × |v|² × (ω̂ × v̂)
//   ρ      = air density (1.225 kg/m³ at sea level)
//   C_L    = Magnus lift coefficient (~0.3 for tennis ball, ~0.25 for golf)
//   A      = cross-sectional area = π × radius²
//   ω̂      = unit spin axis (from 6DOF racket orientation at impact)
//   v̂      = unit velocity vector

void ASimSphereActor::ApplyMagnusForce(float DeltaTime)
{
    if (SpinAxis.IsNearlyZero() || ProjectileMove->Velocity.IsNearlyZero()) return;

    const float   AirDensity  = 1.225f;          // kg/m³
    const float   LiftCoeff   = MagnusCoefficient; // per-game tunable
    const float   Area        = PI * Radius * Radius * 0.0001f; // cm² → m²
    const float   Speed       = ProjectileMove->Velocity.Size() * 0.01f; // cm/s → m/s
    const FVector SpinUnit    = SpinAxis.GetSafeNormal() * SpinRate;
    const FVector VelUnit     = ProjectileMove->Velocity.GetSafeNormal();

    // Cross product gives Magnus direction; magnitude scales with v²
    const FVector MagnusDir   = FVector::CrossProduct(SpinUnit, VelUnit);
    const float   MagnusMag   = 0.5f * AirDensity * LiftCoeff * Area * Speed * Speed;
    const FVector MagnusForce = MagnusDir * MagnusMag * 100.f; // → UE units

    // Add as acceleration to the projectile movement component
    ProjectileMove->Velocity += MagnusForce * DeltaTime / Mass;

    // Spin decays over time from air drag — curve-driven
    SpinRate *= SpinDecayCurve->GetFloatValue(DeltaTime);
}

void ASimSphereActor::Tick(float DeltaTime)
{
    Super::Tick(DeltaTime);

    WaveTime += DeltaTime * WaveSpeed;
    ImpactStrength *= FMath::Pow(Damping, DeltaTime);

    UWorld* W = GetWorld();
    if (!W || !MPC_Sim) return;

    UMaterialParameterCollectionInstance* Inst =
        W->GetParameterCollectionInstance(MPC_Sim);
    if (!Inst) return;

    // 8 scalars → GPU does all per-vertex math via WPO
    Inst->SetScalarParameterValue("Amplitude",      Amplitude);
    Inst->SetScalarParameterValue("Frequency",      Frequency);
    Inst->SetScalarParameterValue("WaveTime",       WaveTime);
    Inst->SetScalarParameterValue("Damping",        Damping);
    Inst->SetScalarParameterValue("ImpactX",        ImpactUV.X);
    Inst->SetScalarParameterValue("ImpactY",        ImpactUV.Y);
    Inst->SetScalarParameterValue("ImpactStrength", ImpactStrength);

    FVector CamLoc; FRotator CamRot;
    if (APlayerController* PC = W->GetFirstPlayerController())
    {
        PC->GetPlayerViewPoint(CamLoc, CamRot);
        FVector ToCam = (CamLoc - GetActorLocation()).GetSafeNormal();

# WaveSolver.py — Huygens wave principle on XDNA2 NPU
import numpy as np, json, pandas as pd

FPS      = 120
DURATION = 30.0
N_KEY    = int(FPS * DURATION)

t = np.linspace(0, DURATION, N_KEY)

# Primary radial wave + ambient breathing
amplitude = 0.8 * np.exp(-t * 0.02) * (1.0 + 0.3 * np.sin(t * 1.5))
frequency = 2.1 + 0.4 * np.sin(t * 0.3)
phase     = np.cumsum(frequency) / FPS
envelope  = np.cos(phase) * amplitude

# Export as DataTable-ready CSV

    "Phase":    phase,
    "Envelope": envelope,
})

// HLSL — wave WPO material custom node 
// Inputs: WorldPos, UV, MPC scalars (Amp, Freq, Time, ImpactX/Y/Str)
// Output: per-vertex world position offset

float3 WavePosition(float3 WorldPos, float2 UV,
                    float Amp, float Freq, float Time,
                    float ImpactX, float ImpactY, float ImpactStr)
{
    float2 ImpactUV = float2(ImpactX, ImpactY);
    float dist     = distance(UV, ImpactUV);
    float radial   = sin(dist * Freq * 6.28318 - Time * 4.0)
                     * exp(-dist * 3.0) * ImpactStr;

    float ambient  = sin(WorldPos.x * Freq * 0.1 + Time)
                    * sin(WorldPos.y * Freq * 0.1 + Time * 1.3) * Amp;

    float3 Normal  = normalize(WorldPos);
    return Normal * (radial + ambient);
}

WPO — World Position Offset. A material output that displaces vertices in world space on GPU.
