void ASimSphereActor::Tick(float DeltaTime)
{
    Super::Tick(DeltaTime);

    WaveTime += DeltaTime * WaveSpeed;
    ImpactStrength *= FMath::Pow(ImpactDecayRate, DeltaTime * 60.f);

    if (bUseBakedDataTable) SampleBakedFrame();
    PushMPCParameters(DeltaTime);

    // Push 8 scalars to MPC — GPU material reads, computes WPO
    MPCInst->SetScalarParameterValue("Amplitude",      Amplitude);
    MPCInst->SetScalarParameterValue("Frequency",      Frequency);
    MPCInst->SetScalarParameterValue("WaveTime",       WaveTime);
    MPCInst->SetScalarParameterValue("ImpactStrength", ImpactStrength);
}

⛨ Failure-Contained by Design · How You Experience It

Every action you take in this editor has a safety net. Errors never block you; they appear as gentle toasts in the bottom-right. State is auto-saved every 30 seconds; you can roll back any change. Four distinct play modes let you validate before you commit, observe before you control, play before you ship. Pre-flight checks turn green before "Play" activates — so you never press a broken button. This is what UEFN feels like. This is what you feel.

▶ The Four Play Modes · Hierarchy of Safety click any to enter that mode · validation gates each transition

① TEST

~3s · NO STATE COMMIT
  • Dry-run validation
  • Spawns nothing
  • Compiles materials
  • Verifies bindings
  • Reports as toast
✓ Always safe to press

② SIMULATE

~5s · OBSERVE-ONLY
  • Physics runs live
  • No player input
  • Camera orbits sphere
  • Frame budget shown
  • Cancel any time · ESC
✓ Cannot break level state
▶ DEFAULT

③ PLAY (PIE)

~8s · FULL INTERACTION
  • Full Play-In-Editor
  • All input devices
  • Editor stays running
  • Hot-reload supported
  • State preserved on stop
✓ Auto-save every 30s

④ RUN (Standalone)

~15s · PACKAGED PROCESS
  • Spawns child .exe
  • Editor isolated · safe
  • Performance accurate
  • OpenXR ready
  • Crash = editor alive
✓ Editor never crashes with the game

▸ Why This Hierarchy Protects You

In every other engine, you have one "Play" button. If it breaks, you don't know if it's your code, your assets, your plugins, or the engine. Here, you climb the ladder: Test passes → you know assets are fine. Simulate passes → physics is fine. Play passes → input is fine. Run passes → packaging is fine. Each stage isolates the failure mode.

The Toast Contract

Errors never open modal dialogs. They never freeze the editor. They never lose your work. Every toast either auto-dismisses (info/warn) or has explicit action buttons (error). You can keep working while issues exist. This is the single most important user-facing reliability promise.
